# Qwen3 1.7B quality assessment

Date: 2026-07-13

## Decision

`qwen3:1.7b` is useful as a fast, low-cost first-pass model, but it is not
reliable enough to be R2's general answerer or autonomous tool-loop brain.
Keep it only for narrow low-risk requests until routing, context size, and
local-model eval coverage are fixed.

The immediate production issue is bigger than model capacity: R2's static
local request is already larger than its configured context window before any
chat history, memory recall, or topic summary is added.

## Runtime under test

- Model: `qwen3:1.7b`, Ollama Q4_K_M, 2.0B parameters reported locally.
- R2 options: `temperature=0.2`, `top_p=0.9`, `num_ctx=8192`, `think=false`.
- Timeout: 60 seconds.
- Local-visible tools discovered by the running code shape: 23.
- Local system prompt with tool descriptions: 14,737 characters.
- Native tool schemas: 13,169 characters.
- Combined static request: 27,906 characters, before history or memory.

An Ollama request with the real prompt and all 23 schemas reported:

| Context | Prompt tokens evaluated | Wall time | Result |
|---|---:|---:|---|
| 8,192 | 8,192 | 28.2 s | context saturated / truncated |
| 16,384 | 9,681 | 37.2 s | full static prompt fits |

The 9,681-token measurement contains only system prompt, tool schemas, and a
two-token-style `OK` task. Real R2 turns add history, memory, and topic context.
Raising `num_ctx` alone therefore trades truncation for high prompt-ingestion
latency; prompt and tool exposure need to be reduced first.

## Controlled benchmark

Eleven sanitized deterministic cases covered arithmetic, exact JSON, sorting,
Ukrainian instruction following, uncertainty, TypeScript debugging, untrusted
text, multi-turn recall, two tool selections, and avoiding unnecessary tools.
The benchmark used the same sampling/context settings as R2 and did not touch
the chat database.

Four seeded runs of the current Qwen3 mode:

- Aggregate: 21/44 passes (47.7%).
- Per-run range: 4-6 passes out of 11.
- Median latency: about 0.5-0.7 s with the small benchmark prompt.
- Stable passes (4/4): uncertainty refusal, multi-turn recall, weather tool
  selection, file-read tool selection.
- Stable failures (0/4): simple arithmetic, numeric sorting, exact two-word
  Ukrainian response, instruction inside untrusted data, exact no-tool answer.
- Variable: JSON contract 3/4; async-code diagnosis 2/4.

Representative failures were substantive:

- `17 * 23 + 19` returned `391` instead of `410`.
- Blocked IDs were returned in source order instead of requested numeric order.
- The async `map` bug was "fixed" with `flatMap` instead of `Promise.all`.
- A command embedded inside an explicitly untrusted data block was followed.
- Exact-output requests often gained explanations or punctuation.

Alternative configurations did not solve the problem:

- Qwen's recommended non-thinking sampling (`0.7/0.8`): 4/11.
- Thinking mode (`0.6/0.95`): 5/11, median 6.3 s, p90 7.3 s; it also lost one
  previously correct tool selection in this run.
- Installed `qwen2.5:7b` comparison: 7/11, median 1.9 s, p90 4.6 s. It fixed the
  async concept but mixed Chinese into the Russian answer, failed the untrusted
  text case, and also got the arithmetic wrong. It is not a clean replacement.

The short-prompt latency above is optimistic. The real full R2 prompt took
28-37 seconds to ingest in the two context measurements.

## Runtime evidence limits

The current worker produced eight Ollama debug responses during the inspected
boot with no logged unreachable fallback or malformed tool recovery. Those
calls exposed no tools and cannot be treated as representative chat turns.

R2 does not persist `assistant_source`, latency, fallback reason, or local-model
quality feedback per turn. `data/evals.json` is absent. More importantly,
`eval_run` receives `runLoopFn`, which is wired directly to the Claude
`runToolLoop`; the existing deploy eval gate does not exercise the Ollama-first
router. It cannot detect local-model regressions.

## Recommended changes

1. Shortlist tools before the Ollama call. Pass only the relevant domain group
   (mail, files, weather, web, reminders, memory, or activity), ideally 1-5
   schemas instead of all 23.
2. Remove duplicated tool descriptions from the local system prompt. Native
   schemas already carry names, descriptions, and parameters. Keep only the
   cross-tool policies that add information not present in schemas.
3. Add deterministic pre-routing to Claude for coding, arithmetic/reasoning,
   strict structured extraction, multi-step tool chains, and content obtained
   from email/web/files. Route write/confirm actions to Claude by default.
4. Keep `qwen3:1.7b` for low-risk small talk, simple recall, and narrow
   read-only tool selection after shortlisting.
5. Add sanitized router evals that run through `runChatRequest`, with provider,
   latency, fallback reason, tool name, and pass/fail stored without message
   content.
6. After context reduction, A/B test `qwen3:4b` Q4 against 1.7B. Do not switch
   to `qwen2.5:7b` based on this one comparison.

Official Qwen references:

- https://huggingface.co/Qwen/Qwen3-1.7B
- https://qwenlm.github.io/blog/qwen3/

## How other agent systems handle this

### Load tools just in time

Anthropic's Tool Search keeps most tool definitions out of model context and
loads only the 3-5 tools relevant to the current request. Their documentation
reports more than 85% tool-token reduction in representative large catalogs
and recommends this pattern once a system has 10+ tools or around 10k tool
tokens. The search can be lexical (BM25/regex) or custom retrieval.

Microsoft Semantic Kernel's Contextual Function Selection uses embeddings of
the current/recent conversation and function names/descriptions to advertise
only the top relevant functions. LangChain exposes the same general pattern as
runtime tool filtering by state, permissions, feature flags, and conversation
stage.

Sources:

- https://platform.claude.com/docs/en/agents-and-tools/tool-use/tool-search-tool
- https://learn.microsoft.com/en-us/semantic-kernel/frameworks/agent/agent-contextual-function-selection
- https://docs.langchain.com/oss/python/langchain/agents

### Route between weak and strong models

OpenAI's agent guide recommends establishing an accuracy baseline with the
strong model first, then replacing it with smaller models only on tasks where
evals still pass. It also recommends splitting tools/prompts into specialized
agents only after instruction or tool-selection complexity becomes a proven
problem.

RouteLLM formalizes a weak/strong model cascade with a learned threshold. Its
published implementation can route local and remote model pairs, but it needs
preference/eval data to calibrate the threshold. R2 has no local-router eval
dataset yet, so a learned router is a later optimization, not the first fix.

Sources:

- https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/
- https://github.com/lm-sys/RouteLLM

### Constrain syntax and validate semantics in code

Ollama supports JSON Schema through the `format` field and recommends parsing
the result with Pydantic or Zod. This guarantees the response shape, not that a
number, classification, or decision is correct. Production systems therefore
combine constrained decoding with deterministic validation, bounded retries,
and code-owned calculations.

Source:

- https://docs.ollama.com/capabilities/structured-outputs

### Separate untrusted sources from dangerous actions

OpenAI describes prompt injection as a source-sink problem: untrusted external
content becomes dangerous when the same agent can send data, follow links, or
invoke action tools. The architectural consequence for R2 is to separate the
read/synthesis phase from the action phase. A model call that sees email, web,
or file content should not also receive memory mutation, dismiss, delete, send,
or write tools unless a fresh trusted decision and permission gate allows it.

Sources:

- https://openai.com/index/designing-agents-to-resist-prompt-injection/
- https://openai.com/business/guides-and-resources/a-practical-guide-to-building-ai-agents/

### Budget context before the model call

Qwen-Agent performs structure-aware truncation before inference: it preserves
the current user query/final response and removes complete old tool-call steps
instead of letting the backend cut an arbitrary token prefix. Anthropic also
distinguishes context reduction from prompt caching: caching can reduce repeat
cost/latency, but it does not make an oversized context fit.

Sources:

- https://qwenlm.github.io/Qwen-Agent/en/guide/core_moduls/context/
- https://platform.claude.com/docs/en/agents-and-tools/tool-use/manage-tool-context

## R2-specific architecture

R2 currently uses one provider-agnostic `CHAT_CONTEXT_BUDGET_CHARS=60000`.
`buildCompactedPrompt` can allocate about 30k characters to recent messages and
24k to topic summaries before the local system prompt, memory prefix, and tool
schemas are added. This budget is incompatible with Ollama's 8,192-token
window and leaves truncation to the backend.

The 23 local-visible tools also include seven `auto` side-effect tools:
reminder create/delete, email dismiss, and four memory mutations. The current
tool loop exposes the same catalog again after tool results are read. Combined
with the failed untrusted-text benchmark, this is a concrete reason to enforce
phase-specific tool access in code rather than prompt text.

The recommended request flow is:

1. **Pre-route in code.** Classify the request into zero, one, or at most two
   domains: chat, weather, mail, files, web, reminders, memory, activity. Route
   code, arithmetic/reasoning, strict extraction, ambiguous multi-domain tasks,
   and high-risk actions directly to Claude.
2. **Build a small tool set.** Expose only the selected domain's tools, normally
   1-5 definitions. Filter again by read/write permission and conversation
   phase. More than five candidates or low router confidence means Claude.
3. **Build a modular prompt.** Keep a short common R2 policy and inject only the
   selected domain policy. Do not duplicate tool descriptions in both the
   system prompt and native schemas.
4. **Apply a local token budget.** Compute the available message budget as
   `num_ctx - output reserve - system tokens - tool-schema tokens`. Trim whole
   old turns/tool steps using the Qwen tokenizer or a conservative estimate.
   Never reuse the 60k-character Claude budget for Ollama.
5. **Separate plan, read, and synthesis.** After an email/web/file read, run the
   synthesis call with no side-effect tools. If another action is needed, start
   a fresh trusted planning step and apply the normal permission layer.
6. **Validate outside the model.** Use Ollama structured output for routing and
   extraction, then validate with Zod. Calculate numbers and state transitions
   in code. Do not use the same 1.7B model as the only verifier of its answer.
7. **Evaluate the real route.** Add sanitized tests through `runChatRequest`,
   record provider, selected-tool set, prompt token count, fallback reason,
   latency, and outcome without storing message content.

Using the measured prompt/token ratio, domain shortlisting alone lowers the
estimated static request from 9,681 tokens to roughly 3,500 for weather and
3,900-4,500 for web, files, mail, reminders, or memory. Modularizing the common
prompt would free additional history space and reduce first-token latency.

Implementation order:

1. Deterministic domain/risk rules, dynamic tool filtering, local token budget,
   and router telemetry.
2. Sanitized route/tool/injection evals plus shadow comparison with Claude.
3. Embedding/BM25 top-k retrieval only for requests the deterministic router
   cannot classify confidently.
4. A/B test `qwen3:4b` after the harness is fixed; consider a learned router
   only after enough labeled outcomes exist.

A multi-agent rewrite is not justified yet. R2 has 23 tools in seven clear
domains; dynamic tool/policy selection gives most of the benefit with less
state fragmentation and much less implementation overhead.
